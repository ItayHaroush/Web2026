# תאריך תחילת פעילות (`owner_activity_started_at`)

## מה זה עושה

שדה nullable בטבלת `restaurants`: **תאריך** שממנו ואילך **המסעדן** רואה דשבורד, רשימת הזמנות, דוחות יומיים, ייצוא CSV/PDF (בהקשר טננט), תובנות AI וכו’ — כאילו לפני התאריך לא היו הזמנות (מסננים לפי `orders.created_at`).

- **אין מחיקת נתונים** — ההזמנות נשארות ב־DB.
- **עריכה רק בסופר־אדמין** — `PUT /api/super-admin/restaurants/{id}` עם `owner_activity_started_at` (או `null` לביטול הסינון).
- **בילינג פלטפורמה / עמלה / חשבוניות** — לא מסוננים; מבוססים על כל ההיסטוריה.

## תפעול

לפני שינוי תאריך בפרודקשן מומלץ **גיבוי DB**.

### בדיקות ידניות

1. מסעדה עם הזמנות לפני ואחרי התאריך — דשבורד מסעדן מציג רק נתונים מהתאריך.
2. `owner_activity_started_at = null` — התנהגות כמו לפני התכונה.
3. התחזות סופר־אדמין (`X-Tenant-ID`) — אותה תצוגה כמו למסעדן.
4. דוח יומי / CSV — רק ימים מהתאריך; גישה לדוח ישן לפי ID מחזירה 404.
5. ייצוא מס (`tax-csv`) — רק הזמנות של אותה מסעדה ומתאריך הסינון.
